# 统一质量门框架

跨所有 skill 共享的质量保障规则。每个 skill 在写入输出文件前必须执行
本文件中的通用门控，再叠加自己的领域专属 self-check。

## 通用质量门（所有 skill 必须执行）

- [ ] 输出文件路径与 `pipeline.md` 中定义的路径完全一致（大小写敏感）
- [ ] 输出文件为有效 UTF-8 编码，非空
- [ ] 所有引用的输入文件都已成功读取（不依赖记忆，每次从磁盘读取）
- [ ] 输出内容不包含 `__MISSING__` 字段（若存在，说明信息不完整，需上报）
- [ ] 输出内容不包含未替换的占位符（如 `{xxx}` 或 `{{field}}`）

## 三次重试协议

```
执行 self-check
  ↓ 通过 → 写入输出文件，继续
  ↓ 失败 → 记录失败原因 → 第2次尝试
              ↓ 通过 → 写入输出文件，继续
              ↓ 失败 → 记录失败原因 → 第3次尝试
                          ↓ 通过 → 写入输出文件，继续
                          ↓ 失败 → 写入带 __MISSING__ 标记的文件
                                   → 向用户明确报告：哪个字段缺失、原因是什么
                                   → 停止当前 skill，等待用户补充
```

**重要**：三次重试都是**针对同一 skill**的不同生成尝试，不是对整个流水线的重试。
orchestrator 在收到 skill 的失败报告后，决定是否允许用户手动补充后继续。

## 上报格式（失败时用户看到的消息）

```
⚠️ [skill名称] 第3次尝试后仍有未通过检查项：

未通过项：
- [ ] {具体检查项描述}
  原因：{为什么无法满足这个条件}
  需要用户提供：{具体需要什么信息}

当前输出已写入（含 __MISSING__ 标记）：{文件路径}
请补充上述信息后，运行 `继续生成` 指令恢复流程。
```

## 输入文件缺失协议

在每个 skill 开始时，先验证所有必需输入文件存在：

```
对每个必需输入文件 F：
  如果 F 不存在：
    停止当前 skill
    报告："{F} 不存在，请先运行 {产生F的skill名称}"
    不要尝试继续或跳过此依赖
```

**不允许**在缺少输入文件的情况下从记忆中"猜测"内容继续执行。
